제 6 회 "관계형 데이터베이스 언어" 강의는 관계형 데이터베이스 언어에 대한 이론적인 순수 언어인 관계대수(relational algebra)와 실용 언어인 SQL을 소개합니다. 여러분들은 다음 사항들을 유념하여 강의해 임해주길 바랍니다.

  • 관계 대수의 기본연산 및 추가연산의 종류를 파악하고 각 연산의 일반구조와 기능을 이해합시다..
  • SQL의 DDL(데이터 정의어)과 DML(데이터조작어)의 종류를 파악하고 각 명령어의 일반형식과 그 기능을 이해합시다.
  • 뷰(view)를 이해하고 뷰를 이용하는데 있어서의 장단점을 알아 봅시다.
  • SQL과 관계대수와의 연관성을 알아봅시다.

 

       데이터 언어(Data Language)란 데이터베이스를 구축하고 관리하면서 데이터베이스 내의 정보를 검색하는데 사용되는 언어로서 사용자와 DBMS(데이터베이스관리 시스템 : DataBase Management System))또는 응용 프로그램(Application Program)과 DBMS 사이에 대화를 하기 위한 수단이다.

       앞의 정의처럼 사용자는 데이터베이스 구축과 관리 또는 검색을 위해 데이터 언어를 직법 이용하여 DBMS와 대화를 할 수 있으며 혹은 데이터 언어 명령문들을 포함한 응용 프로그램을 실행함으로써 DBMS와 대화를 행할 수 있다.

      데이터 언어는 데이터 정의어(Data Definition Language : DDL), 데이터 조작어(Data Manipulation Language : DML), 질의어(Query Language)로 분류된다. 데이터 정의어는 기본적으로 데이터베이스의 논리적 스키마를 기술해 주는 언어이며, 데이터 조작어는 데이터 삽입, 삭제, 갱신을 비롯한 데이터 베이스 관리를 위한 언어이다. 그리고, 질의어는 사용자가 DBMS(데이터베이스 관리 시스템)와 직접적인 인터페이스를 가지면서 데이터 베이스 정의 및 조작을 행할 수 있는 환경을 갖는 언어이다. 이러한 세가지 부류의 언어들은 별도로 존재하는 것은 아니고 이들이 모여서 데이터 언어를 구성하는 것이다. 또한, 질의어는 일반적으로 사용자가 DBMS와 직접 대화할 수 있는 환경을 갖는 특성만 가질 뿐, 데이터 정의와 데이터 조작어 모두를 포함한다. 따라서 용어 상에서 질의어와 데이터 언어를 별도로 구분하지 않고 사용하는 경향이 있다.

      관계형 데이터 언어에서 데이터 정의어는 릴레이션 스키마 및 릴레이션 테이블 생성, 각 릴레이션 스키마 속성의 도메인 선언, 주키이 설정, 외래 키이 선언, 제약조건(constraint) 선언, 뷰(view) 선언(뷰란 제 1 회 강의 내용인 "제 1 회 데이터베이스란 무엇인가 ?"의 데이터 추상화 부분에서 이미 설명한 바 있다)에서 등을 포함하며, 데이터 조작어는 튜플 검색을 비롯한 튜플의 삽입, 삭제, 갱신, 릴레이션 삭제 등의 연산을 포함한다.

      관계형 데이터베이스 시스템에서 데이터 언어는 데이터 언어 자체에 대한 순수한 이론적 접군울 위해 고안된 일종의 형식어(formal language)인 순수 데이터 언어와 상용 데이터베이스 시스템에 포함되어 제공되고 있는 상용 데이터 언어로 구분되는데 일반적으로 순수 데이터 언어에 대한 이론적 접근과 연구는 상용 데이터 언어의 표현력과 질의 최적화(query optimization)(데이터 조작을 위해 표현된 질의어 문(statement)의 처리 속도를 빠르게 하도록 하기 위해 본래의 질의 문과 동일한 결과를 산출하도록 하면서 본래의 질의문 처리 절차를 변환하는 기법을 말한다) 등과 관련된 처리 능력을 확장하는데 직접적인 기초로 활용된다.

      본 강의에서는 몇 개의 대표적인 순수 데이터 언어 중에서 관계 대수(relational algebra)와 실용 데이터 언어인 SQL을 소개하도록 한다. 본 강의에서 순수 데이터 언어인 관계 대수를 소개하는 이유는 관계형 데이터베이스 언어가 수학적으로 잘 정의되어 있음을 보이고자 함과 동시에 여러분들이 SQL 문의 처리과정을 쉽게 이해하는데 도움이 되도록 하기 위함이다.

1. 관계 대수(Relational Algebra)

      관계 대수는 순수 데이터 언어로서 절차적 언어(procedural language)이다. 이 언어는 하나 혹은 두 개의 릴레이션을 입력으로하여 그 결과로서 하나의 새로운 릴레이션을 생성하는 연산들의 모임으로 이루어진다.

      관계 대수는 기본 연산으로 select, project, union, set difference, Cartesian-product, 그리고 rename 등의 연산들을 포함하며, 이 기본 연산들의 조합으로 이루어지는 set interaction, natural join, division, 그리고 assignment 등의 추가 연산들이 존재한다.

    기본 연산

      select 연산

        select 연산은 릴레이션으로부터 주어진 조건을 만족하는 튜플들을 추출하는 기본 연산이다. 이 연산에 대한 일반 형식과 연산 내용을 자세히 소개하면 다음과 같다.

         

        일반형식

         σP (r)

           기호의미

          • σ: select 연산 기호
          • P : 조건
          • r :릴레이션

           연산내용

        주어진 릴레이션 r로부터 조건 P를 만족하는 튜플들을 렐레이션 테이블 형태로 출력한다.

        형 식

        의 미

          A 〉 V

          속성 A의 값이 V보다 클 조건

          A〈  V

          속성 A의 값이 V보다 작을 조건

          A ≤ V

          속성 A의 값이 V보다 작거나 같을 조건

          A = V

          속성 A의 값이 V보다 같을 조건

          A ≠ V

          속성 A의 값이 V보다 다를 조건

        실행 결과

         

        위의 연산은 조건이 두 개(소속학과가 컴퓨터이고(and) 주소가 서울시인 조건)로 형성된 복합 조건이다. 복합 조건은 단일 조건을 두 개 이상 조합하여 생성할 수 있으며 C1과 C2를 각각 단일 조건이라 할 때 이를 조합하는 방법은 다음과 같다.

          C1 ∨ C2

          조건 C1 이 만족되거나 혹은 조건 C2가 만족될 조건

      union 연산

        union 연산은 합집합 연산이다. 즉, union 연산은 릴레이션 스키마가 같은 두 릴레이션의 튜플들을 하나의 릴레이션 테이블 형태로 출력하는 연산이다. 이 연산의 결과로 출력되는 릴레이션 테이블에서 중복된 튜플들은 하나로 취급된다. union 연산의 일반형식과 연산내용, 그리고 연산이 성립될 수 있는 조건은 다음과 같다.

         

        일반형식

         r ∪ s

           기호의미

          • ∪: union 연산 기호
          • r : 릴레이션
          • s :릴레이션

           연산내용

        릴레이션 r과 릴레이션 s에 속한 튜플들을 모두 합친 튜플들로 이루어진 릴레이션 테이블을 출력한다. 즉, 릴레이션 r에 속하거나 혹은 릴레이션 s에 속한 튜플들을 모두 출력한다. 단, 동일한 튜플들은 하나로 취급한다.

          연산조건

        두 릴레이션 r과 s의 릴레이션 스키마가 동일해야 한다. 즉, 속성의 순서 및 개수가 동일하고 각 속성의 도메인이 같아야 한다.

      set difference 연산

        set difference 연산은 차집합 연산이다. 즉, 한 릴레이션으로부터 다른 릴레이션에 공통으로 속한 튜플들을 제거하고 남은 튜플들을 릴레이션 테이블 형태로 출력하는 연산이다. 이 연산에 대한 일반형식과  연산내용, 그리고 연산이 성립될 수 있는 조건을 소개하면 다음과 같다.

         

        일반형식

         r ―s

           기호의미

          • ―: select 연산 기호
          • r : 릴레이션
          • s :릴레이션

           연산내용

        릴레이션 r로부터 릴레이션 s에 속한 튜플들을 뺀 나머지 튜플들로 이루어진 릴레이션 테이블을 출력한다.

           연산조건

        두 릴레이션 r과 s의 릴레이션 스키마가 동일해야 한다. 즉, 속성의 순서 및 개수가 동일하고 각 속성의 도메인이 같아야 한다.

         

        set difference 연산 예

        set difference 연산은 차집합 연산이다. set difference 연산을 설명하기 위해 다음 그림 6-1의 OCU대학생 릴레이션과 한국대학생 릴레이션을 고려하자. 이 두 릴레이션은 같은 릴레이션 스키마를 갖기 때문에 위 표의 연산조건이 만족되므로 set difference 연산이 가능하다. 한국대학생에 소속되어 있지 않으면서 OCU대학생에만 소속되어 있는 학생을 출력하는 질의는 다음의 set difference 연산으로 작성될 수 있다.

      cartesian-product 연산

        cartesian-product 연산은 두 릴레이션간의 카테젼 곱을 수행하여 릴레이션 테이블 형태로 결과를 출력하는 기본연산이다. 즉, 한 릴레이션의 각 튜플에 다른 릴레이션의 각 튜플을 대응시켜서 연결한 튜플들로 구성된 릴레이션 테이블을 출력한다. 이 연산에 대한 일반형식과 연산내용을 소개하면 다음과 같다.

         

        일반형식

         r × s

           기호의미

          • ×: cartesian-product 연산기호
          • r : 릴레이션
          • s : 릴레이션

           연산내용

        릴레이션 r과 릴레이션 s의 카테젼 곱 연산을 수행한다.

        즉, 릴레이션 r의 각 튜플에 릴레이션 s의 각 튜플 을 연결하여 생성한 새로운 튜플들로 구성된 릴레이션 테이블을 출력한다.

        이 연산 결과로 출력된 릴레이션 테이블의 속성들은 두 릴레이션 r과 s의 모든 속성들로 이루어진다.

        일반형식-1

        이 rename 형식은 연산 실행 결과로 만들어진 릴레이션 테이블에 이름을 붙여 출력하는 연산에 대한 형식이다. 이 연산에 대한 일반형식과 연산내용은 다음과 같다.

         

        일반형식-1

         ρn (E)

           기호의미

          • ρ: rename 연산 기호
          • E : 릴레이션 연산식

        일반형식-2

         ρn(A1,A2,...,An) (E)

           기호의미

          • ρ: rename 연산 기호
          • E : 릴레이션 연산식
          • A1,A2,...,An : 속성들

           연산내용

        릴레이션 연산식 E의 실행 결과인 릴레이션 테이블을 n이라는 이름으로 출력하되, 결과 릴레이션 테이블의 속성들을 차례데로 각각 A1,A2,...,An으로 재명명하여 출력한다.

      set-intersection 연산

        set-intersection 연산은 교집합 연산이다. 즉, 이 연산은 두 릴레이션에 공통으로 속한 튜플들을 릴레이션 테이블 형태로 출력한다. 이 연산에 대한 일반 형식과 연산 내용을 자세히 소개하면 다음과 같다.

         

        일반형식

         r ∩ s

           기호의미

          • ∩: set-interaction 연산 기호
          • r : 릴레이션
          • s :릴레이션

           연산내용

        릴레이션 r과 릴레이션 s에 공통으로 속한 튜플들로 이루어진 릴레이션 테이블을 출력한다.

        두 릴레이션 r과 s의 릴레이션 스키마가 동일해야 한다. 즉, 속성의 순서 및 개수가 동일하고 각 속성의 도메인이 같아야 한다.

        r과 s를 각각 두 릴레이션이라 할 때 set-interaction 연산은 기본연산들의 조합으로 다음과 같이 표현될 수 있다. 이에 대한 증명은 강의하고자 하는 초점에 벗어 나므로 생략하기로 한다.

         

        r ∩ s = r - (r - s)

         

      natural-join 연산

        natural-join 연산은 두 릴레이션이 주어질 때 이 두 릴레이션 스키마에서 일치하는 속성들을 중심으로 이 속성 값들이 동일한 두 릴레이션의 튜플들을 연결하여 릴레이션 테이블을 생성한다. 단, 이 결과 릴레이션 테이블 스키마에서 중복된 속성들은 단일화하여 출력된다.

        r(R)과 s(S)를 각가 R과 S를 릴레이션 스키마로 하는 두 릴레이션이라 하고 R ∩ S = {A1,A2,...,An}이라 할 때 이 연산에 대한 일반 형식과 연산 내용을 자세히 소개하면 다음과 같다.

         

        일반형식

         r s

           기호의미

          • : natural-join 연산 기호
          • r : 릴레이션
          • s :릴레이션

           연산내용

        r s =

         ΠR∪S  (σr.A1=s.A1∧r.A2=s.A2∧...∧r.An=s.An (r×s))


        두 릴레이션에서 속성 A1, A2,..., An 값이 일치하는 튜플들을 서로 연결하여 생성한 튜플들을 포함하는 릴레이션 테이블을 출력한다. 연산 결과의 릴레이션 테이블의 릴레이션 스키마는 R∪S이다.

         

        natural-join 연산  예

        natural-join 연산을 설명하기 위해 다음 그림 6-12의 OCU대학생 릴레이션과 OCU교수 릴레이션을 고려하기로 한다.

        OCU대학생 한국대학생

         

        위의 연산은 OCU대학생 릴레이션과 OCU교수 릴레이션에서 공통 속성들인 소속학과, 주소 값들이 같은 튜플들을 짝을지어 릴레이션 테이블 형태로 출력하되 이때의 결과 릴레이션 테이블의 스키마는 중복되는 속성들인 소속학과, 주소 속성들은 하나로 단일화하여 {학번, 이름, 소속학과, 주소, 출신학교, 교수번호, 교수이름}이 된다.

        이 연산 과정을 좀 더 자세히 설명하면 우선 OCU대학생 릴레이션에서 튜플 (001, '손영실', '컴퓨터', '강릉시', '한일')에서 두 릴레이션의 공통 속성인 소속학과 및 주소 값은 각각 '컴퓨터', '강릉시'이며 이 값들을 갖는 OCU교수 릴레이션 튜플은 (002, '이순재', '컴퓨터', '강릉시')이다. 따라서 이 튜플을 연결하여 짝지우고 중복되는 '컴퓨터', '강릉시' 값을 각각 하나씩 남겨서 하나의 튜플로 생성하면 (001, '손영실', '컴퓨터', '강릉시', '한일', 002, '이순재') 튜플이 생성된다. 또한, OCU대학생 릴레이션의 다음 튜플인 (002, '한미자', '컴퓨터', '서울시', '두상')에 대해서도 소속학과와 주소 속성 값인 '컴퓨터'와 '서울시'값을 갖는 OCU교수 릴레이션의 튜플을 찾으면 (004, '박상길', '컴퓨터', '서울시')와 (006, '김성수', '컴퓨터', '서울시')의 두 튜플이 대응되므로 이들을 각각 (002, '한미자', '컴퓨터', '서울시', '두상') 튜플과 짝지어 연결하면 (002, '한미자', '컴퓨터', '서울시', '두상',004, '박상길')과 (002, '한미자', '컴퓨터', '서울시', '두상', 006, '김성수')의 두 튜플이 생성된다. 이러한 방식으로 튜플들을 생성하여 릴레이션 테이블 형태로 나타낸 연산 결과가 다음의 그림 6-13과 같다.

          

        그림 6-13. OCU대학생 한국대학생 의 실행 결과

         

        사실상 위 그림의 natural join 연산은 소속학과가 같으면서 같은 주소를 갖는 학생과 교수에 대한 대응 정보를 구하는 질의 연산이다.

        기본 연산에 의한 natural-join 연산 표현

        r(R)과 s(S)를 각가 R과 S를 릴레이션 스키마로 하는 두 릴레이션이라 하고 R ∩ S = {A1,A2,...,An}이라 할 때 natural-join 연산에 대한 기본 연산들의 조합은 다음과 같다.

         

          r s =

              ΠR∪S  (σr.A1=s.A1∧r.A2=s.A2∧...∧r.An=s.An (r×s))

          

        theta-join 연산

        theta-join 연산은 cartesian-product 연산과 select 연산을 단일 연산으로 결합한 natural-join 연산의 확장이다.

        r(R)과 s(S)를 각각 릴레이션이라 하고 θ를 스키마 R∪S 속성들 상의 조건이라 할 때 theta-join 연산 r θ s 는 다음과 같이 정의된다.

         

              r θ s = σθ(r × s)

         

      division 연산

        r(R)과 s(S)를 각각 릴레이션이라 하고 S⊆R 이라 하자. division 연산은 릴레이션 스키마 R - S 상의 릴레이션 테이블 형태를 출력하는 연산으로 이 연산에 대한 일반형식과 연산내용은 다음과 같다.

         

        일반형식

         r ÷ s

           기호의미

          • ÷: natural-join 연산 기호
          • r : 릴레이션
          • s :릴레이션

           연산내용

        튜플 t 가 r÷s에 속하기 위해서는 다음의 두 조건을 모두 만족해야 한다(즉, t 가 r÷s에 속하기 위한 필요충분 조건)

         1.

        t 는 ΠR-S(r) 의 튜플이다.

        기본 연산에 의한 division 연산 표현

        r(R)과 s(S)를 각가 R과 S를 릴레이션 스키마로 하는 두 릴레이션이라 하고  S⊆R 일때 division 연산에 대한 기본 연산들의 조합은 다음과 같다.

         

        r ÷ s = ΠR-S(r)-ΠR-S((ΠR-S(r)×s)-ΠR-S,S(r))

          

      assign 연산

        assign 연산은 관계 대수 연산의 중간 결과를 릴레이션 변수(relational variable)에 일시적으로 저장하는 추가 연산이다.

        이 연산에 대한 일반형식과 연산내용은 다음과 같다.

         

        일반형식

         t ← E

           기호의미

          • ←: assign 연산 기호
          • t : 릴레이션 변수
          • E :릴레이션

           연산내용

        관계 대수 연산식 E를 수행한 결과의 릴레이션 테이블을 릴레이션 변수 t 에 저장한다.

         

        assign 연산  예

        division 연산에 대한 개념을 쉽게 이해 할 수 있도록 하기 위해 다음의 관계 대수 연산식을 고려하자.

            

        ΠR-S(r)-ΠR-S((ΠR-S(r)×s)-ΠR-S,S(r))

          

        위의 연산식은  division 연산 r ÷ s을 기본 연산들의 조합으로 나타낸 것이다. 이 식을 릴레이션 변수를 이용하는 division 연산을 이용하면 다음과 같이 변환할 수 있다.

            

            temp1 ← ΠR-S,S(r)

            temp2 ← ΠR-S((temp1×s)-ΠR-S,S(r))

            result = temp1  - temp2

         

        위의 지정문(assignment)의 결과는 사용자에게 출력되는 릴레이션 형태는 아니다. 오히려 지정문 기호(←) 오른쪽의 수행결과가 왼쪽의 릴레이션 변수로 할당되며, 이 릴레이션 변수는 후속 연산식에서 사용될 수 있다.

        assign 연산을 사용함으로써 여러개의 연산들로 이루어진 관계 대수 질의는 다수의 assign 연산을 이용하는 절차적 프로그램 형태로 나타낼 수 있다.

2. SQL

      SQL은 관계형 데이터베이스 언어의 국제 표준으로 확립된 비절차적(non-procedural) 언어이다. SQL은 여러 버전들이 존재하는데 원래의 초기 버전은 IBM의 San Jose 연구소(현재는 Amaden Research Center)에서 개발되었다. 이언어는 원래 1970 년대 초에 System R 프로젝트의 일부로 구현되었는데 처음에는 Sequel이라 불리웠다.

      그 이후 Sequel 언어는 더욱 향상되어 SQL(Structured Query Language)로 변경되었고 현재 수 많은 제품들이 이 SQL 언어를 지원하고 있다.

      1986년 미국 표준인 ANSI와 국제 표준인 ISO가 SQL86이라는 SQL 표준을 발표했다. IBM은 1987년 SAA-SQL라는 자체의 SQL 표준을 발표했다.

      1989년 SQL의 확장된 표준인 SQL-89가 발표되었고 오늘날의 데이터베이스 시스템들은 전형적으로 SQL-89 특성들을 최소한  지원한다.

      현재의 ANSI/ISO SQL 표준은 SQL-92 표준이다.

    SQL의 구성

      SQL은 다음의 여러 기능들로 구성된다.

       

      • 데이터 정의 언어(DDL ; Data Definition Language)

        SQL DDL은 릴레이션 스키마 정의, 릴레이션 삭제, 인덱스 생성, 릴레이션 스키마 변경을 위한 명령어를 제공한다.

         

      • 대화식 데이터 관리 언어(DML ; Data Manipulation Language)

        SQL DML은 릴레이션에 튜플 삽입, 삭제, 갱신을 위한 질의어를 제공한다.

            

      • Embedded DML

        SQL의 embedded 형태는 C, Pascal, COBOL 등과 같은 범용 언어 내에서 SQL을 사용할 수 있도록 설계 되었다.

         

      • 뷰(view) 정의

        SQL DDL은 뷰를 정의할 수 있는 명령어를 포함한다.

         

      • 데이터 접근 권한(authorization) 서술 기능

        SQL DDL은 릴레이션과 뷰에 대한 접근 권한을 기술하는 명령어를 포함한다.

         

      • 무결성 제약조건(Integrity Constraint) 서술기능

        데이터베이스에 저장된 데이터가 만족해야 하는 무결성 제약조건을 명령어를 포함하며 이 명령어는 무결성 제약조건을 위반하는 갱신을 허용하지 않는다.

         

      • 트랜잭션(Transaction) 제어

        SQL 은 트랜잭션의 시작과 끝을 기술하기 위한 명령어를 제공하며 동시성 제어를 위한 명시적인 로킹(locking)을 제공한다.

        데이터 검색(select) 명령

        select 명령은 하나 이상의 릴레이션으로부터 조건에 따라 튜플들을 검색하여 주어진 속성값들만 출력하는 명령어이다.

        select 명령어의 기본 형식은 다음과 같다.

         

              select A1, A2, ..., An

              from r1, r2, ..., rm

              where P

        여기에서,

          r1, r2, ..., rm은 각각 릴레이션

           

          A1, ...,An은 각각 릴레이션 r1, r2, ..., rm 상의 속성들

           

          P는 r1, r2, ..., rm들의 속성들 사이에 주어진 조건

         

        select 명령의 실행 결과는 릴레이션 형태를 취한다.

        위의 기본 형식에서 select 명령문은 릴레이션 r1,r2...rm 들로부터 조건 P를 만족하는 튜플들을 찾아서 속성 A1,A2,...,An의 값들로 project 하여 릴레이션 테이블 형태로 출력하라는 의미를 갖는다.

        위의 기본 형식에서 볼 수 있듯이 select 명령문은 select, from, where 의 3절로 구분되는데 각 절의 기능을 살펴보도록 하자.

         

        • select 절은 질의 결과로 출력하고자 하는 릴레이션 형태를 이루는 속성들의 리스트를 갖는 절로 관계 대수의 project 연산에 해당한다.
        • from 절은 데이터 검색에 이용되는 릴레이션들의 리스트를 갖는 절로 관계 대수의 카테젼 곱(cartesian product) 연산과 대응된다.
        • where 절은 데이터 검색 조건을 나타내며 관계 대수 select 연산의 검색 조건에 해당한다.

        위의 select 기본 연산을 관계 대수 연산으로 표현하면 다음과 같다.

          

        ΠA1,A2,...,An(σP(r1×r2×...×rm))

        데이터 삽입(insert) 명령

          insert 명령문은 릴레이션 테이블에 하나 혹은 그 이상의 튜플을 삽입하는 SQL 명령으로 그 일반 형식과 예를 살펴보기로 한다.

          단일 튜플 삽입

          r을 A1, A2,...,An을 속성으로하는 릴레이션이라 하고, V1,V2,...,Vn을 각각 속성 A1, A2,...,An에 대응되는 속성 값들이라 할 때 릴레이션 r에 한 개의 튜플 (V1, V2, ..., Vn)을 삽입하는 insert 명령문의 구조는 다음과 같다.

                   

            insert

            into r [(A1,A2,...,An)]

            values (V1,V2,...,Vn)

           

          위의 일반형식에서  [(A1,A2,...,An)]에서  괄호 []의 의미는 석택적(optional)임을 나타낸다. 사실 (A1,A2,...,An)은 튜플 (V1,V2,...,Vn)의 대응하는 속성값 순서를 나타내는데, Vi은 속성 Ai(i=1,2,...,n)의 값임을 나타내며, (V1,V2,...,Vn)의 속성값 순서가 릴레이션 r의 스키마 속성 순서와 동일하다면 생략 가능하다.

            단일 튜플 삽입 예 

            그림 6-16에서 OCU 대학에 김철수 학생 한명이 편입함으로써 OCU대학생 릴레이션에 튜플 (019, '김철수', '컴퓨터', '강릉시', '강릉고') 를 삽입하는 insert 문은 다음과 같다.

                     

              insert

              into OCU대학생

              values (019,'김철수','컴퓨터','강릉시','강릉고')

               

          복수 튜플의 삽입

          select 문을 이용하여 하나이상의 튜플들을 릴레이션에 삽입하는 명령문의 일반형식은 다음과 같다.

                   

            insert

            into r [(A1,A2,...,An)]

            select문

           

          위의 insert 문은 select문에의해 생성된 튜플들을 릴레이션 r로 삽입하는 명령문이다.   위의 일반형식에서  [(A1,A2,...,An)]에서  괄호 []의 의미는 석택적(optional)임을 나타낸다. 사실 (A1,A2,...,An)은 select문에 의해 생성된 튜플들에 대응하는 속성값 순서를 나타내는데, 속성값 순서가 릴레이션 r의 스키마 속성 순서와 동일하다면 생략 가능하다. 

            복수 튜플 삽입 예 

            그림 6-16에서 한국대학 컴퓨터 학과만 OCU 대학에 통합되었다고 가정하자. 이 때 한국대학 컴퓨터 학과 학생들은 모두 OCU대학으로 편입되어야 하며, 이를 위한 insert 문은 다음과 같다.

             

              insert

              into OCU대학생

              select *

              from 한국대학생 as  S

              where S.소속학과='컴퓨터'

           

        데이터 갱신(update) 명령

          update 명령은 릴레이션의 하나 이상의 튜플을 대상으로 속성값을 갱신하는 SQL 문이다.

          r을 릴레이션이라 하고, V1, V2, ..., Vn을 각각 속성 A1, A2,..., An의 변경해야 할 새로운 값이라 할 때 update 명령문의 일반형식은 다음과 같다.

                     

            update  r

            set A1=V1,A2=V2,...,An=Vn

            [where 조건]

            

          위의 명령문은 where 절의 조건에 해당하는 튜플들을 골라서 속성 A1, A2,..., An의 값을 각각 V1, V2,..., Vn으로 변경한다.

          릴레이션 r의 모든 튜플 값을 변경하고자 할 때에는 where 절을 생략한다.

          튜플 갱신 예

          그림 6-18의 직원 릴레이션 예에서 홍보부가 기획부로 흡수되었다고 가정하면 홍보부 직원들의 소속부서는 모두 기획부로 변경해야 한다. 이에 대한 갱신 질의는 다음과 같다.

                      

            update  직원

            set 소속부서='기획부'

            where 소속부서='홍보부'

            

          select 문을 이용한 갱신 예

           그림 6-16의 OCU대학생 릴레이션과 한국대학생 릴레이션 예에서 OCU 대학생이면서 한국대학생인 학생들이 모두 한국대학의 기숙사로 옮기고  그 기숙사는 서울시에 있다고 가정하자. 이 때 OCU 대학생이면서 한국대학생인 학생들의 주소를 서울시로 모두 변경해야 하는데 이에 대한 질의는 다음과 같다.

                      

            update  OCU대학생

            set 주소='서울시'

            where OCU대학생.주민번호 in

                  (select  한국대학생.주민번호

                   from 한국대학생)

                        

            update  한국대학생

            set 주소='서울시'

            where 한국대학생.주민번호 in

                  (select  OCU대학생.주민번호

                   from OCU대학생)

            

        데이터 삭제(delete) 명령

          delete 명령은 릴레이션에서 하나 이상의 튜플을 삭제하는 SQL 문이다.

          r을 릴레이션이라 할 때 delete 명령문의 일반형식은 다음과 같다.

                     

            delete

            from  r

            [where 조건]

            

          위의 명령문은 where 절의 조건에 해당하는 튜플들을 골라서 릴레이션 r에서 삭제한다.

          릴레이션 r의 모든 튜플을 삭제하고자 할 때에는 where 절을 생략한다.

          튜플 삭제 예

           그림 6-18의 직원 릴레이션 예에서 총부무가 없어지면서 총무부 직원이 모두 사직했다고 가정하면 직원 릴레이션의 튜플들 중에서 소속부서 속성값이 '총무부'인 튜플들을 모두 삭제해야 한다. 이에 대한 질의는 당ㅁ과 같다.

           

            delete

            from  직원

            where 소속부서 = '총무부'

            

          select 문을 이용한 삭제 예

            그림 6-16의 OCU대학생 릴레이션과 한국대학생 릴레이션 예에서 OCU 대학생이면서 한국대학생인 학생들이 모두 OCU대학을 마치고 한국대학에만 재학 중이라면, OCU대학생이면서 한국대학생인 학생들을 OCU대학생 릴레이션으로부터 제거해야 한다. 이에 대한 질의는 다음과 같다.

                      

            delete

            fromt  OCU대학생

            where OCU대학생.주민번호 in

                  (select  한국대학생.주민번호

                   from 한국대학생)

                  

        뷰(view) 생성

          뷰(view)란 ?

          뷰(view)란 데이터베이스에 존재하는 하나 이상의 릴레이션들을 가지고 각 릴레이션의 일부 또는 전부를 이용하여 새롭게 생성된 일종의 릴레이션 테이블을 말한다. 뷰는 일반 릴레이션과 똑 같이 select 문에 이용될 수 있다. 그러나, insert, update, delete 문에서는 제한적으로 사용된다.

          뷰는 데이터베이스 내부에서 뷰를 구성하는 튜플들을 갖지 않는다. 단지, 데이터베이스 내부에서는 뷰를 구성하는데 이용된 기존 릴레이션들간의 뷰 생성 관계만 저장되어 있다가 뷰를 이용한 검색이나 조작 연산이 이루어질 때 이 뷰 생성 관계를 이용하여 비로서 뷰의 튜플을 구성할 따름이다.

          뷰 생성 명령

           뷰는 이미 존재하는 릴레이션들을 바탕으로 생성되는데, v를 뷰라고 하고 A1,A2,...,An을 v를 구성하는 속성이라 한다면 뷰 v 생성을 위한 일반 형식은 다음과 같다.

            

            create view  v [(A1,A2,...,An)]

                   as select문     

           

          위의 뷰 선언에서 select문에 의해 생성된 릴레이션 테이블이 뷰 v를 구성하게되며, 데이터베이스 내부에서는 select문의 내용만 포함하고 있다가 뷰가 이용될 때 이 select문이 실행되어 뷰의 튜플들을 구성한다.

          뷰 생성의 예

           뷰 생성을 위한 몇가지 예를 들어보자.

          우선, 그림 6-16에서 OCU대학생 릴레이션과 한국대학생 릴레이션을 union하여 주민번호, 이름, 소속학과 속성을 갖는 OCU_한국대학생 뷰를 생성하고자 한다면 다음의 뷰 생성 DDL을 사용한다.

            

          create view  OCU_한국대학생(주민번호, 이름, 소속학과)  as

            (select 주민번호, 이름, 소속학과 

             from OCU대학생)

            union

            (select 주민번호, 이름, 소속학과 

             from 한국대학생)

           

          앞에서도 설명한 것처럼 OCU_한국대학생 뷰 생성 시에는 튜플이 생성되지 않고 select문에 의한 OCU대학생 릴레이션과 한국대학생 릴레이션 사이의 뷰 생성 관계만 저장되어 있다가 다음과 같은 질의가 실행될 때 이 관계를 실행함으로써 뷰를 구성하는 튜플들이 생성된다.

             

            select

            from  OCU_한국대학생

            where 소속학과 = '컴퓨터'

            

          또 한가지 예로 OCU대학생 릴레이션을 이용하여 OCU컴퓨터학생 뷰를 생성하는 DDL은 다음과 같다.

             

            create view  OCU컴퓨터학생  as

              select *

              from OCU대학생

              where 소속학과 = '컴퓨터'

            

          위의 create view 문에서는 OCU컴퓨터학생 뷰의 스키마가 지정되지 않았다. 이 때에는 select 문에 의해 생성된 릴레이션 테이블의 스키마인 {주민번호, 이름, 소속학과, 주소, 출신학교}를 갖게 된다.

          이번에는 OCU대학생 릴레이션을 이용하여 OCU전자공학생 뷰를 생성하는 DDL은 다음과 같다.

             

            create view  OCU전자공학생  as

              select *

              from OCU대학생

              where 소속학과 = '전자공학'

            

          마지막으로 OCU컴퓨터학생 뷰와 OCU전자공학생 뷰를 이용하여 OCU컴퓨터_전자공학생 뷰를 생성하면 다음과 같다.

                              

          create view  OCU컴퓨터_전자공학생  as

            (select * 

             from OCU컴퓨터학생 )

            union

            (select *

             from OCU전자공학생)

           

          뷰 삭제 명령

          v를 뷰라 할 때 v를 삭제하는 DDL의 일반형식은 다음과 같다.

           

                drop view  v  restrict   

                  혹은

                drop view  v  cascade 

           

           앞에서 생성한 뷰의 예에서 다음의 뷰 삭제 명령을 고려하자.

           

              drop view  OCU_한국대학생  restrict   

           

          OCU_한국대학생 뷰는 다른 뷰 생성에 참조되고 있지 않기 때문에 삭제된다. 그러나, 다음의 뷰 삭제 명령을 고려하자.

           

              drop view  OCU컴퓨터학생  restrict   

           

          OCU컴퓨터학생 뷰는 OCU컴퓨터_전자공학생 뷰  생성에서 참조되고 있기 때문에 restrict 구에 의해 삭제가 취소된다.그러나, 다음의 cascade 구를 이용하여 삭제하면 OCU컴퓨터학생 뷰는 물론 이를 참조하여 생성된 모든 뷰 즉, OCU컴퓨터_전자공학생 뷰도 연쇄적으로 삭제된다.

           

              drop view  OCU컴퓨터학생  cascade

            

          뷰 사용 상의 장점과 단점

          뷰는 데이터베이스 추상 수준의 뷰 레벨에서 존재한다. 뷰를 생성하고 이용하는 이유는 뷰가 제공하는 몇가지 장점을 제공하기 때문인데 이러한 장점을 살펴보면 다음과 같다.

            

          • 뷰는 논리적 데이터 독립성을 어느 정도 제공한다. 뷰 정의에 사용된 릴레이션들이 확장된다던지 혹은 뷰가 속한 데이터베이스에 릴레이션이 더 늘어난다 하더라도 기존의 뷰를 사용하는 응용 프로그램이나 사용자는 아무런 영향을 받지 않는다.
          • 뷰는 데이터 접근을 제어하게 함으로써 보안을 제공한다. 뷰를 통해서만 데이터를 접근하게 하면 뷰에 나타나지 않는 데이터를 안전하게 보호할 수 있다.
          • 사용자의 데이터 관리를 수월하게 해준다. 필요한 데이터를 뷰로 정의해서 처리할 수 있기 때문에 관리가 쉽고 명령문이 간단해 진다.
          • 여러 사용자의 상이한 응용이나 요구를 지원해 준다. 하나의 릴레이션을 가지고 서로 다른 여러개의 뷰를 정의하여 사용자의 요구에 따라 활용할 수 있도록 할 수 있다.

          이러한 장점 외에도 다음과 같은 단점도 존재한다.

             

          • 뷰는 그 정의를 변경할 수 없다.
          • 뷰는 삽입, 삭제, 갱신 연산에 많은 제한을 갖고 있다.
